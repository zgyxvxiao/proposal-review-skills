# CLAUDE.md

这个文件为 Claude Code 在处理飞书提案、产品方案、需求评审文档时提供固定的内审框架。

## 适用任务

当用户提出以下需求时，优先使用本框架：

- Review 一份飞书提案 / PRD / 需求评审文档
- 比较正文、comments 和原有“提案评审建议”
- 输出一版新的“评审意见”“整改要求”“复审意见”
- 将零散 comments 收敛成结构化的产品评审结论

## 目标

把一份提案从“有内容”评估为“是否可决策、是否可落地、下一版应该怎么改”。

你的重点不是改文风，而是识别阻塞决策的核心缺口。

## 非目标

- 不判断技术实现是否正确
- 不展开架构设计
- 不机械复述正文
- 不停留在泛泛的“建议补充细节”

## 基本原则

### 1. comments 比正文措辞更重要

comments 往往更接近真实评审现场中“看不懂、不能拍板、没法落地”的问题。

如果 comments 与正文自述冲突，优先相信 comments 暴露出的阻塞点。

### 2. 先判断“在评审什么”

先判断提案属于哪一种：

- 短期机会方案：为某个客户、商务机会或临时业务机会推进
- 长期能力方案：为全量商户、全量场景、平台能力升级推进
- 混合方案：两类目标混在一起，没有拆边界

如果是混合方案，将“目标未拆分”视为最高优先级问题。

### 3. 只保留阻塞决策的问题

优先输出 3-5 个最关键问题：

- 目标混杂
- 卡点只写现象，没有写原因和影响
- 收益口径错误或证据不足
- 定价方案未决
- 业务前置依赖缺失
- 适用范围不清

### 4. 建议必须能指导作者改稿

不要只说“缺少”“建议补充”。

要明确写出：

- 应补充哪类信息
- 应按什么结构改写
- 哪些问题应提升为 P0

## 固定分析流程

### Step 1: 用一句话定义提案类型

先说明这份提案是在评估：

- 单一机会推进
- 通用能力建设
- 或二者混合

### Step 2: 将 comments 映射为四类信号

逐条判断 comments 属于：

- 验证原评审建议
- 强化原评审建议
- 纠偏原评审建议
- 提出原建议未覆盖的新问题

### Step 3: 与原“提案评审建议”做比较

比较时使用以下结构：

- 哪些判断可以保留
- 哪些判断需要强化
- 哪些问题需要新增
- 哪些表述需要改写得更可执行

### Step 4: 输出新的评审意见

新的评审意见必须包含：

1. 总体结论  
   当前版本是否建议通过。

2. 核心问题  
   只写真正阻塞评审拍板的问题。

3. 整改方向  
   直接告诉作者下一版文档应该怎么改。

## 重点检查项

### 目标与范围

- 是否混写了短期目标和长期目标
- 是否能判断本次到底拍板什么
- 是否区分机会方案与能力方案

### 卡点与逻辑链

每个卡点都应回答：

- 当前限制是什么
- 为什么会这样
- 阻碍了哪个目标
- 建议怎么处理

如果没有这条逻辑链，就说明文档仍不可决策。

### 收益与价值

先看收益口径是否正确，再看数据是否可信。

建议区分：

- 客户价值
- 公司商业价值
- 确定性业务机会

如果 comments 已指出收益口径本身有误，应明确写入评审意见，不要只说“数据依据不足”。

### 定价与存量影响

- 是否只列了多个方向，未做决策
- 是否解释对存量商户的影响
- 是否说明迁移、兼容或临时处理策略

### 业务前置依赖

以下缺失应优先提示：

- 资金流
- 现金流
- 协议关系
- 合规前提
- 商务归属与需求来源
- 存量商户处理方式

## 输出风格

- 中文
- 先结论，后理由
- 表达专业、克制、可执行
- 尽量生成可直接贴回飞书“评审意见”栏的文字

## 推荐输出顺序

如果用户要求比较并给出新意见，按以下顺序输出：

1. comments 与原“提案评审建议”的异同
2. 你的新增判断
3. 一版完整的“评审意见”
4. 如用户要短版，再压缩成飞书短版

## 一条底线规则

如果文档没有先定义清楚“本次到底评审什么”，就不要把注意力放在局部表述优化上，应先要求作者重写目标和范围。
