# Good Review Examples

这份文件不是模板库，而是给 reviewer 一个“什么叫高质量判断”的参照。

优先在以下场景读取：

- 你已经判断出问题，但想把话说得更像正式评审意见
- 你需要把 comments、正文、原评审建议合并成一版更强的结论
- 你不确定应该先打“目标不清”还是先打“收益口径错误”

## Example 1: 目标混杂型

适用场景：

- 一份提案同时写了短期机会和长期能力建设
- 评审人看完后不知道这次到底在拍板什么

高质量写法：

> 当前提案混合了短期机会方案和长期能力开放方案，两者评审边界、改造复杂度与决策重点均不同。建议先拆分评审对象，否则无法判断本次结论应针对单一客户机会，还是通用产品能力建设。

为什么这段写法有效：

- 先下总体判断
- 直接指出“为什么不能评”
- 改稿建议是“拆分评审对象”，不是空泛地说“建议补充”

常见低质量写法：

> 目标写得不够清楚，建议明确。

问题：

- 太轻
- 没有说明“不清楚”具体指什么
- 没告诉作者下一版怎么改

## Example 2: 结构完整但口径错误型

适用场景：

- 文档章节齐全
- 但收益、价值、成功指标的定义本身有偏差

高质量写法：

> 当前文档并非只有收益测算依据不足的问题，更关键的是收益口径本身需要重定义。若本次能力升级的核心价值在于提升大额出款服务能力或降低 FX 成本，则不应继续以通用手续费增长作为主要论证依据。

为什么这段写法有效：

- 区分了“证据不足”和“定义错误”
- 把讨论从数字可信度拉回到价值定义
- 能防止作者只去补一个更漂亮的增长数字

## Example 3: 渠道能力差异型

适用场景：

- 同一支付方式背后接了多个渠道
- 失败问题持续出现，但大家只在修特殊路由

高质量写法：

> 当前问题的本质不是单一错误码异常，而是同一支付方式下，不同渠道对目标机构、收款字段和格式校验的要求不一致，但现有支付能力未将这些差异显式建模，导致成功率问题只能通过临时路由和人工兜底处理。

为什么这段写法有效：

- 没有停留在“某渠道不稳定”
- 把问题从渠道层提升到产品能力表达层
- 直接指向长期方案方向：能力建模

## Example 4: 可直接落入飞书评审意见的结构

推荐顺序：

1. 第一段写总体结论  
   当前不建议直接通过 / 建议补充后复审

2. 第二段写 P0 问题  
   优先写“评审对象不清”“收益口径错误”“能力未建模”这种阻塞问题

3. 第三段写支撑理由  
   用 2-4 条关键事实支撑，不要复述全文

4. 最后一段写整改方向  
   写成“建议补充/明确/重写什么”

示例句式：

> 建议下一版优先完成三项整改：一是统一评审对象与方案范围；二是补齐问题、限制条件与解决方向的逻辑链；三是明确收益口径及商户影响，避免继续以临时路由作为主要解决方案。

## 使用提醒

- 这些示例的重点是判断结构，不是措辞本身
- 不要照抄全文，优先复用其判断顺序和句式力度
