# Domain Checklists

这份文件给支付、出款、渠道接入类提案一个更细的检查清单。

优先在以下场景读取：

- 本地出款 / wallet / local banks / bank transfer
- 新增渠道 / 替换渠道 / 主备切换 / 分流切换
- 商户字段、目标机构、账户类型、手机号规则相关问题

## Checklist 1: 出款能力升级

至少检查：

- 本次是新增渠道、能力补齐，还是能力标准化
- 到账时效变化是否有真实对比基线
- 限额提升是否涉及目标机构差异
- 收款要素是否变化
- 对商户是否需要改入参

## Checklist 2: 渠道接入 / 多渠道并存

至少检查：

- 同一支付方式下有几个渠道
- 渠道是否真等价，还是只看起来等价
- 目标机构支持范围是否一致
- 字段映射是否一致
- 格式校验是否一致
- 是否已有主备、分流、回切、特殊路由

如果文档大量依赖这些词，通常说明“能力未建模”。

## Checklist 3: 收款字段

至少检查：

- `account` 和 `phone` 是否并存
- 如果并存，谁优先
- 不同目标机构分别消费哪个字段
- 字段是否需要标准化过滤
- 是否存在国家码 / 区号问题
- 是否存在银行账号与手机号需一致的规则

## Checklist 4: 目标机构

至少检查：

- 是否所有渠道都支持同一个目标机构集合
- 某目标机构在一个渠道里是银行还是钱包
- 目标机构映射是否跨渠道一致
- 是否存在“同一收款对象在不同渠道映射到不同机构”的情况

## Checklist 5: 账户类型与证件

至少检查：

- 账户类型是否强制要求 `S/C` 或 `saving/checking`
- 某些机构是否要求固定账户类型
- 证件类型和证件号是否跨渠道规则一致
- 是否已有明确失败码和样例单支撑

## Checklist 6: 商户影响

至少检查：

- 签约商户数
- 有交易商户数
- SKA / KA / 普通商户分布
- 重点客诉商户是谁
- 若改商户入参，需要通知多少商户
- 若不改商户，内部是否能完全消化

## Checklist 7: 止血方案 vs 长期方案

短期止血常见形式：

- 特殊路由
- 原单重发
- 主备切换
- 回切某渠道
- 给重点商户加白名单

长期方案常见形式：

- 渠道能力建模
- 上游字段标准化
- 前置格式校验
- 目标机构能力标准化
- 自动路由决策

如果文档只有止血手段，没有长期方案，评审意见里要明确指出。

## Checklist 8: 最终评审时必须能回答的 5 个问题

1. 这次到底在评审什么  
2. 哪些商户真的受影响  
3. 商户现在到底送了什么字段  
4. Dlocal / Thunes / 其他渠道具体哪里不一样  
5. 这次改完后，是止血，还是从能力上解决了问题
