# Proposal Review For Feishu

这个 skill 用于把一份提案从“能看懂”提升到“能决策、能改稿”。

## 适用场景

- 用户让你 review 飞书提案、PRD、需求评审文档
- 用户提供正文、comments、已有“提案评审建议”，要求你比较后给出新的判断
- 用户要求输出可直接粘贴回飞书的“评审意见”“整改要求”“复审意见”
- 用户希望把 comments 里的零散反馈，整理成结构化的评审结论

## 不做什么

- 不评判技术实现是否正确
- 不展开系统架构设计
- 不把重点放在措辞润色，而是放在产品决策质量
- 不机械复述正文；重点提炼阻塞决策的缺口

## 核心判断原则

1. comments 的优先级高于正文措辞  
   comments 往往反映真实评审现场中“看不懂、不能拍板、无法落地”的阻塞点。

2. 先判断评审对象是否清晰  
   优先识别提案是在评估：
   - 单一客户 / 单一商机的短期机会方案
   - 面向全量商户或全量场景的通用能力建设
   - 两者是否被混写在一起

3. 先看能不能决策，再看写得漂不漂亮  
   只要存在关键决策前提缺失，就不要因为文档“结构完整”而高估质量。

4. 建议必须可改稿  
   不只指出“缺什么”，而是明确“下一版文档该补什么、怎么改写”。

## 高频缺口类型

- 目标混杂：短期机会方案与长期能力建设混在一起
- 论证断层：只写现象，没有写原因、影响和处理方向
- 收益口径错误：价值定义错了，或证据不足
- 定价方案未决：列了方向，但没有明确推荐和影响分析
- 适用范围不清：到底服务谁、开放给谁、限制到什么程度
- 前置依赖缺失：资金流、现金流、协议关系、合规前提、存量商户处理
- comments 指出的理解障碍：方案名字、章节表达、术语和逻辑关系读者无法直接理解

## 建议的分析流程

如果命中下面场景，再按需读取对应 reference：

- 需要找更稳的评审表达、参考过往高质量写法时：读 `references/good-review-examples.md`
- 需要快速判断这份提案属于哪类常见问题模式时：读 `references/review-patterns.md`
- 需要按支付/渠道/商户影响做更细颗粒度检查时：读 `references/domain-checklists.md`
- 需要提醒自己避免“看起来写了很多，但其实不能评”的误判时：读 `references/anti-patterns.md`

### Step 1: 快速归类提案类型

先用一句话定义这份提案究竟是什么：

- 机会型方案：为某个客户、商务机会或单一业务诉求推进
- 能力型方案：为全量商户、全量场景、通用产品能力升级推进
- 混合型方案：两者都写了，但边界没拆开

如果是混合型方案，优先把“目标未拆分”列为 P0 问题。

### Step 2: 抽取 comments 的真实信号

不要只把 comments 当补充说明，要判断每条 comments 属于哪一类：

- 对原评审建议的验证：说明原建议方向对，但还不够落地
- 对原建议的强化：说明问题比原建议描述得更严重
- 对原建议的纠偏：说明原建议没抓住真正问题
- 原建议未覆盖的新问题：需要新增到新版评审意见里

### Step 3: 对原“提案评审建议”做保留/强化/补充/改写

逐条判断：

- 保留：方向正确，仍应出现在新意见中
- 强化：原判断对，但级别要提高
- 补充：原文没覆盖，但 comments 已经暴露出新问题
- 改写：原文表述对但不够可执行，需要改成更像整改要求

### Step 4: 生成新评审意见

新版意见应包含三个层次：

1. 总体结论  
   当前版本能否直接通过，还是应暂缓 / 有条件通过 / 重写后复审。

2. 核心问题  
   只写真正阻塞决策的 3-5 个问题，避免面面俱到。

3. 明确整改方向  
   每条都落到“下一版文档该补什么”。

## 重点检查清单

### 1. 目标与范围

- 这次评审到底在拍板什么
- 是否同时写了短期和长期目标
- 是否明确 In Scope / Out of Scope
- 是否区分当前机会方案和后续平台能力建设

### 2. 卡点与论证链条

每个卡点至少回答四个问题：

- 当前限制是什么
- 形成原因是什么
- 阻碍了哪个目标
- 建议如何处理

如果只描述“存在问题”，没有解释为什么影响目标，则判为论证断层。

### 3. 价值与收益

优先看收益口径是否定义正确，再看数据是否可信。

至少区分：

- 客户价值：客户为什么需要
- 公司价值：收入、成本、margin、机会规模
- 确定性机会：是否有真实客户场景和交易承诺

如果 comments 明确指出收益口径写错，应直接上升为 P0。

### 4. 定价与存量影响

- 是否只列出多个方向，但没有推荐结论
- 是否评估了对已签约商户、不同报价模型商户的影响
- 是否说明迁移原则、兼容原则或临时策略

### 5. 业务前置依赖

以下任何缺失，都可能构成 P0：

- 资金流
- 现金流
- 协议关系
- 商务归属与需求来源
- 合规前提
- 存量商户处理方式

## 输出模板

### A. 比较结论

按下面结构输出：

- 原“提案评审建议”哪些判断成立
- comments 强化了哪些问题
- comments 新增了哪些问题
- 你的最终判断和原建议相比，有哪些升级或调整

### B. 新版评审意见

写成适合飞书的正式表达：

- 第一段：总体结论
- 第二段起：核心问题和原因
- 最后一段：建议的整改方向和评审结论

### C. 若用户需要更短版本

将新版评审意见压缩成 200-300 字，保留：

- 当前不建议直接通过
- 3 个最关键问题
- 下一步如何整改

## 语言要求

- 用中文
- 直接、专业、少空话
- 多用“建议补充说明”“建议明确”“建议重写为”
- 尽量写成可直接进入飞书“评审意见”栏的语言

## 一条实用规则

如果你发现文档的问题不是“没写清楚”，而是“没定义清楚在评审什么”，就优先要求作者先重写目标和范围，不要急着讨论定价、收益或排期。
